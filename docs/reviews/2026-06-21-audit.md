# Dunst MCP Cycle Audit - 2026-06-21

Scope: full repository audit from the current `main` checkout.

Method: `cli-cycle` routed the project through code quality, topology, semantic x-ray, documentation, doc-code sync, tests, shell, wizard/setup, pipeline, GitHub health, diagrams, tree, README, drift, and limited infrastructure checks. Local gates were executed before scoring; GitHub Actions state was inspected through `gh`. The scorecard below is the pre-remediation audit snapshot.

## Executive Summary

The local repository is substantially healthier than the previous full-cycle baseline: the Rust workspace is split into navigable crates and modules, local formatting, clippy, tests, rustdoc, shell, docs, build, and security audit pass. The dependency graph has no duplicate crate families and the codebase has moved away from the earlier monolith shape.

The project is not converged because the remote CI on `main` is red. The last ten GitHub Actions runs inspected were failures. Two concrete blockers explain the current red state:

1. CI pins Rust/Cargo 1.80, while the lockfile currently resolves `clap_derive 4.6.1`, which requires Cargo support for `edition2024`.
2. The Linux pure job compiles `dunst-vision` on Ubuntu, including `vision_bench`, which imports macOS-gated modules without a non-macOS fallback.

These are not theoretical quality issues: they are active gate failures on `main`. Fixing them is the next useful step before another broad refactor.

Overall status: 7.6/10, not Phoenix-converged.

## Same-Day Remediation

After this audit, the Tier 3 CI blockers were addressed in the working tree:

- `.github/workflows/ci.yml` now follows the Grob-style stable Rust toolchain policy instead of pinning Rust 1.80.
- `Cargo.toml` now declares `rust-version = "1.85"`, the first Rust release with stable Edition 2024 support.
- `crates/dunst-vision/src/bin/vision_bench.rs` now mirrors `shapes_bench.rs` with a non-macOS fallback `main`, so Linux can compile the target without macOS-only imports.
- Stale CI push branches were removed from the workflow trigger.
- `crates/dunst-graph/Cargo.toml` no longer declares unused `serde` and `serde_json` dependencies.
- The former Python Markdown link checker was retired in favor of the Grob-style `Docs Lint` workflow: `markdownlint-cli2` plus `lychee`.

Post-fix local validation:

- `cargo fmt --all -- --check`
- `cargo clippy --workspace --all-targets --all-features -- -D warnings`
- `cargo test --workspace --all-targets --locked`
- `cargo check -p dunst-core -p dunst-graph -p dunst-vision --tests --target x86_64-unknown-linux-gnu --locked`
- `cargo machete`
- `npx --yes markdownlint-cli2 --config .markdownlint.json 'docs/**/*.md' README.md`
- `lychee --cache --max-cache-age 1d --no-progress --accept 200,204,206,301,302,303,307,308,403,429 docs/**/*.md README.md`
- `cargo build --release -p dunst-mcp --locked`

The cross-target `cargo test --target x86_64-unknown-linux-gnu --no-run` could not complete on macOS because the host linker is not configured for Linux binaries. The non-linking `cargo check --tests` passed for the same packages and target, which validates the original unresolved-import failure mode.

## Tier 3 - Critical

### 1. CI Rust toolchain no longer matches the lockfile

Evidence:

- `.github/workflows/ci.yml:35`, `.github/workflows/ci.yml:80`, `.github/workflows/ci.yml:133`, `.github/workflows/ci.yml:161` pin `toolchain: 1.80`.
- `Cargo.toml:15` declares `rust-version = "1.80"`.
- `crates/dunst-mcp/Cargo.toml` depends on `clap = "4.6.1"`.
- GitHub run `27884849516`, job `Rust checks / Clippy`, fails while parsing `clap_derive v4.6.1` because `edition2024` support is required.
- Local validation passed on `cargo 1.95.0` / `rustc 1.95.0`, so the local environment hid the CI failure.

Impact: every push to `main` can stay red even when local checks pass. Required checks cannot be trusted.

Recommended fix: choose one policy and encode it consistently:

- Raise the workspace and CI toolchain to a Rust/Cargo version compatible with the current lockfile.
- Or keep true MSRV 1.80 and pin/downgrade dependencies so `cargo test --locked` works with Rust/Cargo 1.80.

Validation:

- Run the chosen toolchain locally with `cargo +<toolchain> clippy --workspace --all-targets --all-features -- -D warnings`.
- Re-run GitHub Actions and confirm `Rust checks` is green.

Sources: `cli-forge-pipeline`, `cli-forge-github`, `cli-audit-code`, `cli-audit-sync`, `cli-audit-test`.

Effort: low to medium.

### 2. Linux CI compiles a macOS-only vision binary

Evidence:

- `.github/workflows/ci.yml:94` runs `cargo test -p dunst-core -p dunst-graph -p dunst-vision --locked` on Ubuntu.
- `crates/dunst-vision/src/lib.rs` gates `capture` and `ocr` behind `#[cfg(target_os = "macos")]`.
- `crates/dunst-vision/src/bin/vision_bench.rs:8` imports `dunst_vision::capture` and `dunst_vision::ocr` unconditionally.
- GitHub run `27884849516`, job `Linux pure checks / Test pure crates`, fails with unresolved imports.

Impact: the pure Linux job is structurally red. This also weakens confidence that platform boundaries are encoded correctly.

Recommended fix: mirror the pattern already used by `shapes_bench.rs`: provide a `#[cfg(not(target_os = "macos"))]` fallback `main` or gate the binary target so it is not compiled on Linux.

Validation:

- Run `cargo test -p dunst-core -p dunst-graph -p dunst-vision --locked` on Linux, or in the GitHub job.
- Confirm `Linux pure checks` is green.

Sources: `cli-forge-pipeline`, `cli-audit-code`, `cli-audit-test`, `cli-audit-tangle`.

Effort: low.

## Tier 2 - Major

### 3. Local and CI toolchain parity is not enforced

Evidence:

- Local gates passed with a newer toolchain.
- CI pins Rust 1.80.
- No `rust-toolchain.toml` was found to make local validation default to the same compiler as CI.

Impact: contributors can get a clean local run while `main` fails remotely.

Recommended fix: after deciding the MSRV policy, add a `rust-toolchain.toml` or a documented `just`/scripted gate that uses the same toolchain as CI.

Sources: `cli-audit-sync`, `cli-forge-pipeline`, `cli-audit-test`.

Effort: low.

### 4. Public API still exposes legacy `VisualOps` naming

Evidence:

- `crates/dunst-core/src/lib.rs:1` now says "Shared contracts for Dunst MCP".
- `crates/dunst-core/src/lib.rs:34` defines `pub enum VisualOpsError`.
- `crates/dunst-core/src/lib.rs:55` aliases `Result<T>` to `VisualOpsError`.

Impact: documentation and package naming have converged on Dunst, but the public core error type still carries the old project name. This is small but visible semantic drift for downstream users and agents reading the API.

Recommended fix: either rename to `DunstError` with a compatibility alias, or explicitly document that `VisualOpsError` is a legacy compatibility name.

Sources: `cli-audit-sync`, `cli-audit-drift`, `cli-audit-code`.

Effort: medium.

### 5. Setup and doctor are useful but not a complete config lifecycle

Evidence:

- `/Users/ludwig/.cargo/bin/dunst-mcp setup --help` exposes client selection and dev wrapper mode.
- `/Users/ludwig/.cargo/bin/dunst-mcp doctor` checks the binary, MCP command, local config files, wrapper, platform, Accessibility, and Screen Recording state.
- The current setup path still prints snippets instead of owning apply/edit/migrate/idempotent config flows.

Impact: onboarding quality is better than a raw README, but still requires manual transfer and can drift between Codex, Claude, and local wrapper states.

Recommended fix: add an explicit config lifecycle: `setup --apply`, `setup --dry-run`, `setup --edit`, `setup --migrate`, and an idempotent recap of the exact files changed.

Sources: `cli-audit-wizard`, `cli-audit-doc`, `cli-audit-sync`.

Effort: medium.

### 6. Test suite lacks deeper semantic-change detectors

Evidence:

- `cargo test --workspace --all-targets --locked` passes 150 tests locally.
- No `proptest`, `quickcheck`, `insta`, `cargo-mutants`, `loom`, or `trybuild` usage was found.
- `cargo test --workspace --doc --locked` runs zero doctests.

Impact: the current tests cover known behavior well, but they are less effective at catching drift in graph invariants, risk policy, tool schemas, and cross-platform cfg boundaries.

Recommended fix: add targeted property/snapshot tests for scene graph and affordance invariants, plus a small mutation-test lane for pure crates.

Sources: `cli-audit-test`, `cli-audit-drift`, `cli-audit-xray`.

Effort: medium.

### 7. Performance has a benchmark lane but no enforced regression baseline

Evidence:

- `.github/workflows/ci.yml:121` defines scheduled/manual benchmarks.
- `.github/workflows/ci.yml:147` runs the graph pipeline benchmark.
- No baseline artifact, threshold comparison, or performance trend file is enforced by CI.

Impact: the benchmark can produce signal, but it does not prevent a regression or make perf changes reviewable over time.

Recommended fix: store benchmark artifacts and compare p50/p95 against a checked-in or previous-run baseline before claiming speed gains.

Sources: `cli-forge-perf`, `cli-forge-pipeline`, `cli-audit-test`.

Effort: medium.

### 8. Repository protection and ruleset observability are weak

Evidence:

- `gh api repos/azerozero/dunst/branches` reports only `main`, with `protected: false`.
- `gh api repos/azerozero/dunst/rulesets` returns 403: rulesets require a public repo or GitHub Pro for this repository.
- `gh pr list --state open` returns no open PRs.
- The last ten GitHub Actions runs inspected were failures.

Impact: there is no visible guard preventing a red `main`, and the configured ruleset surface cannot currently be audited.

Recommended fix: enable available branch protection on `main`, or document the accepted risk if plan limits prevent rulesets. Once CI is green, require the Rust, Linux, shell, and docs jobs.

Sources: `cli-forge-github`, `cli-forge-pipeline`.

Effort: medium.

### 9. Documentation gates check links but not prose quality

Pre-remediation evidence:

- The main CI had a custom Python local-link checker.
- No markdown style, spelling, or terminology drift gate was found.

Impact: docs are useful today, but stale wording and terminology drift can re-enter without a failing gate.

Recommended fix: add a lightweight prose gate such as markdownlint plus a small project dictionary for terms like Dunst, AX, SkyLight, and MCP.

Sources: `cli-audit-doc`, `cli-audit-sync`, `cli-forge-doc`.

Effort: low to medium.

## Phoenix Decision

The cycle should not continue into broad architecture slicing yet. The next Phoenix step is one of these:

1. Fix critical CI blockers now: toolchain/lockfile policy plus `vision_bench` cfg gating.
2. Defer CI and improve code quality: address `VisualOpsError`, unused dependencies, and setup lifecycle.
3. Improve tests first: property/snapshot/mutation tests for pure crates and schema/risk invariants.
4. Improve docs/README first: prose gate, demo artifact, and doc examples.
5. Stop here: keep the report as the current health baseline.

Recommended choice: option 1. A red `main` invalidates the strongest local quality signal.
